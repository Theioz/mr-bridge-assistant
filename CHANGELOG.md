## [Unreleased]

### Fixed
- **Bridge no longer stalls mid-turn after announcing a next action (#369).** Added a system-prompt rule requiring Bridge to either call the corresponding tool in the same turn or end with an explicit confirmation question when it writes a promissory statement ("Now assigning X", "Next I'll...", "Moving on to..."). Root cause was the per-step confirmation cadence (calendar pre-flight, delete/update guards) being over-generalised to every tool call in a bulk-approved batch. Prompt-only fix; no agent-loop or TypeScript changes.


### Added
- **Per-user Google OAuth refresh tokens — multi-tenant Google integrations (#390).** Replaces the single global `GOOGLE_REFRESH_TOKEN` env var with per-user rows in a new `user_integrations` Supabase table. Each user connects their own Google account via `/settings → Integrations` ("Connect Google" button → Google consent screen → callback stores the encrypted refresh token). New `encrypt_integration_token` / `decrypt_integration_token` SQL functions wrap `pgp_sym_encrypt` / `pgp_sym_decrypt` keyed by a new `ENCRYPTION_KEY` env var — first use of column-level encryption in this codebase. `getGoogleAuthClient` is now async and per-user; falls back to the `GOOGLE_REFRESH_TOKEN` env var for the owner until the seed script is run post-deploy (zero-downtime migration). All 12 `getGoogleAuthClient` call sites updated: chat tools (`calendar.ts` 4 sites, `gmail.ts` 2, `workouts.ts` 2) and API routes (`/api/google/calendar`, `/api/google/calendar/upcoming-birthday`, `/api/google/gmail`). Google Fit sync helper (`getGoogleAccessToken`) similarly migrated — drops `GOOGLE_FIT_REFRESH_TOKEN` and `GOOGLE_FIT_CLIENT_ID/SECRET` env vars. OAuth start route at `/api/auth/google/start` and callback at `/api/auth/google/callback` (greenfield — no prior auth-code flow existed). Disconnect deletes the row via a settings server action. API routes now return `{ not_connected: true, status: 403 }` instead of a generic 500 when `GoogleNotConnectedError` is thrown. One-shot seed script at `scripts/seed-owner-google-integration.ts` moves the owner's existing token into the DB post-deploy. **Post-deploy manual steps:** (1) add `ENCRYPTION_KEY` to Vercel env; (2) add redirect URIs to Google Cloud Console OAuth client; (3) run `npx tsx scripts/seed-owner-google-integration.ts`. Closes #390, unblocks #380.

- **Anthropic extended thinking wired into the Sonnet chat path behind `ENABLE_THINKING` flag, default off (#339).** Measurement-first per the plan — the issue's three open decisions (enable at all? UX surface? per-model or global?) needed real data before committing to a UX. PR #350 cleared the way by collapsing the chat surface onto `ToolLoopAgent`, so the config now lives at the agent constructor in [web/src/app/api/chat/route.ts](web/src/app/api/chat/route.ts): `providerOptions: { anthropic: { thinking: { type: "adaptive" } } }` is spread onto `new ToolLoopAgent({...})` only when `modelTier === "sonnet" && process.env.ENABLE_THINKING === "1"` — Haiku and the Groq demo path receive nothing. **Adaptive mode, not enabled+budgetTokens** — per the @ai-sdk/anthropic docs, "for newer models (claude-sonnet-4-6, claude-opus-4-6, and later), use adaptive thinking" so the model self-selects whether each turn warrants reasoning. The existing `tokenBudgetExceeds(TOKEN_BUDGET = 150_000)` predicate in [route.ts](web/src/app/api/chat/route.ts) covers reasoning automatically because Anthropic counts thinking as output tokens. Reasoning is **session-only** by three independent guarantees: (1) `sendReasoning: false` on `toUIMessageStreamResponse` — reasoning bytes don't cross the wire even though the SDK default is `true`; (2) the `chat_messages` insert path persists `text.trim()` only (or the #319 `synthesizeFallbackSummary` output when text is empty), neither of which can include reasoning since they're separate fields on the v6 `OnFinishEvent`; (3) the #319 synthesizer's input is `(steps, flags)` — reasoning isn't an addressable input, so it cannot substitute for a final answer. No client renderer changes; `getMessageText` in [chat-interface.tsx](web/src/components/chat/chat-interface.tsx) keeps filtering on `type === "text"`. **SDK finding** — `@ai-sdk/anthropic@3.0.0` does not break out reasoning into `outputTokens.reasoning` (always `undefined` per [convert-anthropic-messages-usage.ts](web/node_modules/@ai-sdk/anthropic/src/convert-anthropic-messages-usage.ts) lines 66–70 — Anthropic's API doesn't separately report reasoning_tokens in `usage`, the SDK just lumps everything into `outputTokens.total`). So usage-based accounting cannot tell us whether thinking fired. Instead `onFinish` reads `StepResult.reasoning` (an `Array<ReasoningPart>`) and `StepResult.reasoningText` directly — the only definitive signal — and the `[chat] cache ...` log line extends with three new key-value fields: `reasoningParts=<n>` (count of reasoning blocks across all steps), `reasoningChars=<n>` (sum of reasoning text length, a proxy for thinking volume since Anthropic bundles it into output token billing), and `thinkingEnabled=<true|false>` (the gate flag itself), so a baseline-vs-thinking A/B run is grep-able from one stdout file. **Measured A/B (5 prompts, Sonnet 4-6, real Bridge state, single-session-per-arm fallback after the new-chat-button hydration mismatch broke per-prompt fresh sessions):**

  | # | Prompt | Baseline ms | Thinking ms | Δ ms | Baseline out tok | Thinking out tok | reasoningChars | Quality call |
  |---|---|---:|---:|---:|---:|---:|---:|---|
  | 1 | `hi` | 1745 | 2093 | +348 | 12 | 42 | 51 | same |
  | 2 | Friday move + errands | 22177 | 24066 | +1889 | 586 | 1294 | 1746 | baseline (caught Giants game conflict, asked timing, offered Sunday) |
  | 3 | `move my workout` (ambig) | 8538 | 17278 | +8740 | 289 | 770 | 954 | thinking (flagged Giants game as timing constraint) |
  | 4 | deload swap | 20461 | 20553 | +92 | 596 | 1062 | 1289 | tied / baseline (better history narration + side-by-side table) |
  | 5 | meal Q&A | 14103 | 26888 | +12785 | 616 | 1430 | 2093 | **baseline (tool-use regression — thinking said "no saved recipes match" but baseline correctly found Air Fryer Chicken Thighs in saved recipes)** |

  **Aggregates:** mean latency baseline 13.4s → thinking 18.2s (+4.8s, +36%); mean output tokens baseline 420 → thinking 920 (+500, +119%); cost delta ≈ +$7.50 per 1k turns at Sonnet's $15/MTok output rate (cache reads unchanged); reasoning fired on every turn including chitchat (adaptive mode is permissive, not selective). **Subjective:** baseline ≥ thinking on 4 of 5 prompts, with one outright regression where thinking either skipped or misread the `get_recipes` tool. **Recommendation: do not enable by default.** The infrastructure (gate, adaptive provider config, reasoning instrumentation, sendReasoning gate, the SDK-extraction finding) ships behind `ENABLE_THINKING=0` so a future re-measurement can flip the flag — when a new Sonnet snapshot lands, when we want to evaluate Haiku, or when the prompt distribution shifts toward more multi-step planning. Out of scope and deferred unless future data justifies: a UX surface for reasoning (collapsed disclosure or "Bridge is thinking…" ambient strip extending #341), thinking on the Haiku path, swapping back to `enabled` mode with explicit `budgetTokens`, persisting reasoning to `chat_messages`, the per-prompt-conditional gate (route thinking only via the `selectModel` keyword/length predicates). ### Fixed (chat — verified-success contract for mutating tools, issue #319; supersedes #223)
- **Canonical `{ ok, error? }` shape on every state-mutating chat tool** in [web/src/app/api/chat/route.ts](web/src/app/api/chat/route.ts). New `ok()` / `err()` helpers wrap the return values of `add_task`, `complete_task`, `log_habit`, `update_profile`, `create_calendar_event`, `update_calendar_event`, `delete_calendar_event`, `assign_workout`, and `update_workout_exercise`. Previously each tool used a different shape on success (`{ updated: true }`, `{ deleted: true }`, raw row, `{ written: [...] }`) and `{ error: ... }` on failure — so the model and the fallback synthesizer could not reliably distinguish "the action happened" from "the action didn't happen", which let Bridge claim a calendar move had succeeded when the underlying tool result was a hallucination. Every Supabase write now also null-checks the returned row and returns an error if zero rows landed (closes the silent-`.single()`-returns-null class).
- **Read-after-write verification on calendar mutations** for `create_calendar_event`, `update_calendar_event`, and `delete_calendar_event`. After Google's response we issue an `events.get` and confirm the resulting `start` / `end` / `summary` / `location` (or `status === "cancelled"` for delete) match the requested change. Mismatches return `ok: false` with a specific error so Bridge tells the user instead of falsely confirming. The same pattern is applied inline to `update_workout_exercise` (re-reads the workout plan row and confirms the matched exercise's fields contain the requested updates) — addresses the parallel #239 false-success class.
- **Synthesizer fallback inspects tool results, not just tool calls.** `synthesizeFallbackSummary` now pairs each `toolCall` with its `toolResult` via `toolCallId`, splits into `succeeded` / `failed`, and emits one of three branches: "Done. I {phrases}." / "I tried to {phrases} but it didn't go through — {error}. Please try again." / "Partial: I {ok phrases}, but failed to {failed phrases} ({first error}). Check before retrying." This was the structural defect that let PR #247's fallback emit "Done. I updated a calendar event." for a tool that had returned an error.
- **System-prompt verified-success rule** added to `STATIC_SYSTEM_PROMPT`: "Never say a mutating action is 'done' or use ✓ unless the most recent tool result for that action has `ok: true` in this turn. If you don't see a successful tool result, either run the tool now or tell the user the action didn't go through." Last-line-of-defense complement to the structural fixes above.
- **Per-turn deadline guard so `onFinish` always runs.** New `TURN_DEADLINE_MS = 80_000` and `maxDuration` raised 60 → 90 seconds. The deadline timer aborts the agent loop ~10s before Vercel's hard kill so `onFinish` always fires, persists a fallback assistant message, and emits the SSE turn-complete frame — the exact silent-stall path that hit on issue #319's repro session, where the Lambda died mid-stream and PR #247's fallback never got the chance to run. The client-side `useChat` reads `data` for a `{ type: "turn-complete" }` frame; if absent when the stream ends — i.e. the Lambda was killed without `onFinish` running, or the network cut the SSE early — the parent ([web/src/components/chat/chat-page-client.tsx](web/src/components/chat/chat-page-client.tsx) `handleMessageSent`) refetches the active session from the server. The server-persisted assistant message (verified-success or fallback) becomes the source of truth, replacing whatever partial state useChat held locally.
- **Context loader off-by-direction fixed.** [web/src/app/api/chat/route.ts](web/src/app/api/chat/route.ts) was loading the FIRST 10 messages of a session for context (`order ascending + limit 10`); for any session past 10 turns the model lost recent context including its own prior tool results, which made the bug-1 hallucination easier. The next turn's "Yes, done!" message was generated in 3 seconds with no tool call (proven by row timestamps and the absent assistant-row gap), and the user manually moved the calendar event afterward.

### Fixed (chat agent loop — issue #223)
- **Bridge no longer ends a turn silently.** [web/src/app/api/chat/route.ts](web/src/app/api/chat/route.ts) `onFinish` previously skipped the DB write when the model produced no text (which happens when the AI SDK loop exhausts `maxSteps` on a tool step). - **Diagnostic log on every turn completion.** `[chat] turn complete session=… steps=N/20 tokens=… durationMs=… finishReason=… hitStepCap=… budgetExceeded=… synthesized=…` so we can see step-cap hits and fallback synthesis in production logs.
